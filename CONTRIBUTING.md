# Contributing to rvsc-tools

Thanks for your interest in improving rvsc-tools. Please read this before opening
a PR.

## This project is read-only, by design

rvsc-tools reads and decodes Victron VEConfigure `.rvsc` files. It does not, and
will not, write or modify them. **PRs that add write/modify/export capability
will be declined**, regardless of how well they are implemented.

The reason is safety, not scope-creep for its own sake: a malformed or
misunderstood configuration written back to a MultiPlus/Quattro can damage
batteries or other connected equipment. Victron's own VEConfigure and VRM
Remote VEConfigure validate what they write; this project does not implement
that validation and has no intention of trying to reproduce it. If you need to
change a device's settings, use those vendor tools.

## Running the tests

The test suite uses Python's stdlib `unittest` — not `pytest`. From the repo
root:

```
python3 -m unittest discover tests -v
```

`tests/fixture.rvsc` is synthetic and regenerated by the test suite itself; you
do not need a real device or a real `.rvsc` file to run or extend the tests.

## `core/settings.json` is the source of truth

`core/settings.json` is the single source of truth for the setting/flag/enum
tables used by both the CLI (`rvsc.py`) and the web viewer (`docs/index.html`).
`docs/index.html` is generated from it by `tools/build_web.py`, so the two must
always be regenerated and committed together:

```
python3 tools/build_web.py
```

CI runs `tools/build_web.py --check` and fails the build if `docs/index.html`
is stale relative to `core/settings.json`. If you touch `core/settings.json`,
run the generator and commit the resulting `docs/index.html` diff in the same
PR.

## House style: terminology

Documentation and commit messages in this repository never use the terms
"reverse engineer", "decompile", "disassemble", or "crack". Use "format
analysis" or "derived from publicly distributed software" instead. This is a
deliberate, consistent choice across the project — please follow it in PRs and
issue text as well as code comments.

## Absolute rule: no serial numbers, no real config files

Never commit a device serial number or a real `.rvsc` configuration file — not
in code, tests, issue reports, or commit messages. The only `.rvsc` file that
belongs in this repository is `tests/fixture.rvsc`, which is entirely
synthetic (built byte-by-byte by the test suite, not derived from any real
device). CI scans for both a device-serial pattern and any additional `.rvsc`
file and will fail the build if either is found.

## Help widen hardware coverage (the highest-value contribution)

This is the most valuable thing a contributor can do right now.

The parser's format handling is currently verified against exactly **one**
device: a MultiPlus 24/1200/25-16, firmware 2667558, VEConfigure 1.33.
Everything else is expected to work, because the file format is
self-describing rather than dependent on a per-model lookup table (see
[FORMAT.md](FORMAT.md) §5) — but "expected" is not "confirmed."

If you have different hardware (a different MultiPlus/Quattro model, a
different firmware version, or a different VEConfigure version), you can help
without ever sharing your file:

1. Run the CLI against your own `.rvsc` file, locally:

   ```
   python3 rvsc.py show /path/to/your/file.rvsc
   ```

2. Compare the decoded output against what VEConfigure shows for the same
   file — in particular:
   - the sections listed and the `alignment` score/line (does it parse
     cleanly and report a high in-range percentage?)
   - a handful of setting values you know from VEConfigure (e.g. charge
     current limits, voltage set points, any enum-valued settings you can
     cross-check against VEConfigure's dropdown labels)
3. Open a "config file report" issue (see the issue template) with your device
   model, firmware version, VEConfigure version, whether it parsed, and
   whether the values matched. **Do not attach the `.rvsc` file itself** — it
   contains your device serial number and site-specific configuration.

### Pinning unmapped setting indices

Some setting indices are still unmapped to a human-readable name/meaning. You
can help pin one down with a minimal, controlled change:

1. In VEConfigure, change exactly **one** field/setting.
2. Save the result to a new `.rvsc` file (keep your original untouched).
3. Run:

   ```
   python3 rvsc.py diff old.rvsc new.rvsc
   ```

4. The setting index (or indices) reported as changed is the mapping for that
   field. Open an issue or PR reporting the field's name (as shown in
   VEConfigure) and the index rvsc.py reported for it — this lets us add it to
   `core/settings.json` with a verified label.

As with config reports, do not attach either `.rvsc` file; the index and field
name from the `diff` output is all that's needed.
